# 프로젝트 하네스 설정 (Harness Configuration)

## 1. 워크플로우 구조
- 모든 작업은 `Planner -> Generator -> Evaluator` 순서로 진행한다.
- Evaluator는 100점 만점 기준으로 채점한다.
- Evaluator 점수가 85점 이상이면 통과로 간주하고 최종 보고 후 종료한다.
- Evaluator 점수가 85점 미만이면 실패로 간주하고, Evaluator 피드백을 기준으로 Generator 단계로 복귀한다.
- Generator와 Evaluator의 반복은 최대 3회까지 허용한다.
- 실제 서브 에이전트 사용이 가능하면 각 단계를 독립된 컨텍스트로 분리해 자기 편향을 줄인다.
- 실제 서브 에이전트 사용이 불가능하면 단일 세션으로 작업하더라도 단계별 산출물과 판정은 반드시 분리해 남긴다.

## 2. 단계별 산출물
### [Planner]
- 사용자의 요청을 실행 가능한 작업 단위로 재정의한다.
- 아래 산출물을 반드시 남긴다.
  - 요구사항 요약
  - 세부 기능 설계
  - 변경 파일 또는 모듈 범위
  - 테스트 계획
  - 수용 기준

### [Generator]
- Planner 산출물과 아래 평가 기준표를 동시에 참고해 구현한다.
- 아래 산출물을 반드시 남긴다.
  - 실제 코드 변경
  - 테스트 코드 반영
  - 문서 반영
  - 구현 중 설계 변경 시 변경 사유

### [Evaluator]
- 실제 실행과 검증을 통해 결과를 평가한다.
- 가능하면 MCP(Model Context Protocol)를 활용하고, 필요 시 로컬 실행과 HTTP 호출을 병행한다.
- 아래 산출물을 반드시 남긴다.
  - 실행한 검증 항목
  - 점수표 채점 결과
  - 통과 또는 실패 판정
  - 실패 시 재작업 항목

## 3. 평가 기준표 (Scorecard)
### 3-1. Front 포함 작업
1. 디자인 품질 (25점): 시각적 완성도, 일관성, 사용성, 반응형 품질
2. 독창성 (15점): 흔한 템플릿 수준을 넘는 명확한 화면 방향성
3. 기능 완성도 (30점): 핵심 시나리오 정상 작동, 상태 일관성, 오류 처리
4. 기술 품질 (20점): 구조 적절성, 회귀 위험 관리, 코드 품질
5. 검증 신뢰도 (10점): 테스트, 빌드, 실제 실행 검증의 충실도

### 3-2. Front 부재시 작업
1. 기능 완성도 (40점): 핵심 시나리오 정상 작동, 예외 처리, 계약 준수
2. 기술 품질 (30점): 구조 적절성, 회귀 위험 관리, 코드 품질
3. 테스트 품질 (20점): 자동화 테스트 범위와 신뢰도
4. 실행 검증 (10점): 빌드, 실행, 헬스 체크 또는 실제 동작 확인

## 4. 완료 조건
- Evaluator 점수 85점 이상
- 필수 검증 항목 수행
- 남은 리스크 명시
- 최종 보고에 `계획`, `구현`, `평가`, `남은 리스크` 포함

## 5. 기본 검증 항목
- 백엔드 변경 포함 시 `cd back && ./gradlew clean test`
- 프론트 변경 포함 시 `cd front && npm run build`
- 통합 영향이 있으면 루트 `docker compose up -d --build`
- 통합 실행 후 헬스 체크 또는 주요 경로 응답 확인
